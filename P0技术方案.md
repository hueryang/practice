# 智能安全邮箱 — P0 轻量技术方案

**依据**：[P0需求范围.md](./P0需求范围.md)（单实例、注册/登录/发信/收信闭环；附件与高级安全非 P0 交付项）。

---

## 1. 架构（文字描述）

- **进程边界**：一个 **`server` 进程**（常驻，监听 TCP，如 `:8080`）；多个 **`client` 进程**（用户每次在终端执行子命令或一次交互会话，无状态，不持久化业务数据）。
- **调用方向**：`client` 仅通过 **HTTP/1.1 + JSON** 调用 `server`（实现成本最低、易用 `curl`/脚本验收）。若后续换 gRPC，语义不变，仅换传输层。
- **服务端内部分层**（自上而下）：
  - **接入层**：HTTP 路由、请求体解析、统一错误码与 JSON 响应。
  - **鉴权**：除 `POST /register`、`POST /login` 外，其余需 **Bearer Token**；token 由登录签发，内存或表内可校验（见下文会话策略）。
  - **业务层**：用户注册/登录校验、发信（校验收件人存在、写 `Mail`）、收信（按当前用户查收件箱、读单封）。
  - **存储层**：单库单文件 **SQLite**（或等价：嵌入式 KV + 自建索引；P0 推荐 SQLite 减少胶水代码）。**所有持久化经存储层**，便于日后换存储实现。
- **数据流**：注册/登录写 `User`；发信写 `Mail`（绑定发件人、收件人）；收信只读当前用户为收件人的 `Mail`。P0 **不产生 `Attachment` 记录**，表结构可建可不建，见 §3。

## 2. 模块划分

### 2.1 Server

| 模块 | 职责 |
|------|------|
| `main` / `config` | 监听地址、DB 路径、JWT 密钥或 session 密钥等启动参数（环境变量或配置文件二选一即可）。 |
| `transport/http` | 路由注册、中间件（JSON、`Authorization` 解析）、404/415 处理。 |
| `handler` | `Register`、`Login`、`SendMail`、`ListInbox`、`GetMail` 入参校验与调用 service。 |
| `service` | 密码哈希与比对（如 bcrypt）、用户名唯一性、发信时解析收件人 ID、列表/详情权限（只能读自己的收件箱）。 |
| `store` | `User` / `Mail`（及可选 `Attachment`）的 CRUD；事务：发信时保证 `Mail` 插入原子性。 |
| `session` | 登录后签发 **opaque token**（随机串 + 服务端表或内存映射 `token → user_id, expiry`）或 **JWT**（无状态，P0 任选其一；单进程 JWT 更简单）。 |

### 2.2 Client

| 模块 | 职责 |
|------|------|
| `main` / `cmd` | 子命令：`register`、`login`、`send`、`inbox`、`read`（或交互菜单调用同一套逻辑）。 |
| `config` | `SERVER_URL`（默认 `http://127.0.0.1:8080`）、登录后 **token 文件路径**（如 `~/.mail-cli/token`，仅本机权限 0600）。 |
| `api` | 封装 HTTP：封装上述 REST，处理 JSON 与 `Authorization` 头。 |
| `ui` | 终端打印列表/正文；错误信息直接 stderr。 |

## 3. 数据模型

约定：主键均为 **64 位整型自增**（或 UUID 字符串，二选一全项目统一）；时间统一 **UTC RFC3339** 存库可用 `INTEGER` Unix 秒或 `TEXT` ISO8601。

### 3.1 User

| 字段 | 类型 | 约束 / 说明 |
|------|------|----------------|
| `id` | INTEGER / UUID | 主键 |
| `username` | TEXT | **UNIQUE**，登录名，大小写策略在实现里固定一种（建议存库前统一小写） |
| `password_hash` | TEXT | 仅存 **bcrypt/argon2** 等单向哈希结果，不存明文 |
| `created_at` | DATETIME / INTEGER | 创建时间 |

### 3.2 Mail

| 字段 | 类型 | 约束 / 说明 |
|------|------|----------------|
| `id` | INTEGER / UUID | 主键 |
| `from_user_id` | FK → User.id | 发件人 |
| `to_user_id` | FK → User.id | 收件人（P0 仅站内用户，不存在则发信失败） |
| `subject` | TEXT | 主题，允许空串 |
| `body` | TEXT | 正文，纯文本 |
| `created_at` | DATETIME / INTEGER | 发信时间 |

**索引**：`(to_user_id, created_at DESC)` 利于收件箱列表；`(from_user_id)` 留给后续发件箱（P0 可不查）。

### 3.3 Attachment（P0 预留）

P0 需求不包含附件收发；模型可先定义，**实现阶段可不建表**，或建表始终 0 行。

| 字段 | 类型 | 约束 / 说明 |
|------|------|----------------|
| `id` | INTEGER / UUID | 主键 |
| `mail_id` | FK → Mail.id | 所属邮件；删除邮件时策略后续定（P0 不涉及） |
| `filename` | TEXT | 展示用原始文件名 |
| `content_type` | TEXT | MIME，如 `image/png` |
| `size_bytes` | INTEGER | 字节数 |
| `storage_path` | TEXT | 服务端磁盘相对路径或对象键（单实例本地路径即可） |
| `created_at` | DATETIME / INTEGER | 上传完成时间 |

## 4. 核心 API（必要接口）

**前缀**：`/api/v1`（便于日后版本迭代）。  
**Content-Type**：`application/json`。  
**鉴权**：`Authorization: Bearer <token>`（登录返回的 `token`）。

| 方法 | 路径 | 鉴权 | 请求体 / 说明 | 响应要点 |
|------|------|------|----------------|----------|
| POST | `/register` | 否 | `{"username":"string","password":"string"}` | 201 + `{"user_id"}`；409 用户名冲突 |
| POST | `/login` | 否 | `{"username":"string","password":"string"}` | 200 + `{"token":"string","expires_at":"RFC3339"}`；401 失败 |
| POST | `/mails` | 是 | `{"to_username":"string","subject":"string","body":"string"}` | 201 + `{"mail_id"}`；404 收件人不存在 |
| GET | `/mails` | 是 | 查询参数可选：`limit`（默认 50）、`offset`（默认 0） | 200 + `{"items":[{"id","from_username","subject","created_at"},...]}`（列表不含正文，减少流量） |
| GET | `/mails/{id}` | 是 | — | 200 单封详情含 `body`；403 非该邮件收件人、404 不存在 |

**错误格式**（建议统一）：`{"error":"short_code","message":"human readable"}`，HTTP 状态码 4xx/5xx 与之一致。

**client 侧映射**：

- `register` → POST `/register`
- `login` → POST `/login`，写 token 文件
- `send` → POST `/mails`，读 token
- `inbox` → GET `/mails`
- `read <id>` → GET `/mails/{id}`
