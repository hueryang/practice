# 会议纪要图片报告生成 — 整体方案

## 背景与目标

**场景**：智能会议产品以纯文本输出会议纪要，篇幅过长时影响用户快速获取关键信息。通过图片报告提供更直观、简洁的阅读体验。

**输入**：一份会议纪要文本文件（纯文本，可扩展 Markdown）。

**目标**：

- 提取核心信息：会议主题、关键结论、待办事项（action items）；参与人可选。
- 生成一张结构清晰、布局合理的 PNG 图片报告，保存到本地。

**约束**：技术栈不限；可与 AI 配合完成分析、设计与实现；可调用外部 API。

---

## 一、系统设计（模块划分）

| 模块 | 职责 | 说明 |
|------|------|------|
| **CLI / 入口** | 参数解析、配置加载 | 输入文件路径、输出路径；可选：模板名、是否调用 LLM、API Key 等 |
| **输入层** | 读文件、编码检测、轻量预处理 | 统一为 UTF-8 字符串；可选：去 BOM、合并多余空行；若支持 Markdown，可在此做标题/列表粗解析，或原样交给抽取层 |
| **信息抽取层** | 产出结构化 `MeetingReport` | **方案 A**：LLM（推荐，纪要格式多变）；**方案 B**：规则 + 关键词（无网/无 API 时降级） |
| **渲染层** | 结构化数据 → PNG | 布局（边距、行高、换行）、字体、配色、分区绘制；Python 推荐 **Pillow（PIL）**；Go 可用 `freetype` + `image` 或 `gg` 等 |
| **输出层** | 写磁盘 | 确保目录存在、确定文件名；可选 DPI/尺寸 |

**语言建议**：优先 **Python**（Pillow 成熟、中文排版与换行资料多、与各类 LLM API 集成简单）。若团队以 Go 为主且要求全栈 Go，也可实现，但字体与排版成本略高。

## 二、数据结构设计

核心是一个**与 UI 无关的中间表示**，渲染层只依赖它，便于更换模板或更换抽取实现。

### 2.1 核心模型

```text
MeetingReport
├── title: string              # 会议主题（展示用标题）
├── summary: []string          # 关键结论，每条一句或一小段
├── action_items: []ActionItem
└── participants: []string     # 可选；题目未强制，建议预留
```

```text
ActionItem
├── content: string            # 待办描述
├── owner: string | null       # 可选：负责人
└── due: string | null          # 可选：截止时间（保留原始字符串即可）
```

### 2.2 辅助结构（实现时可放在渲染模块内部）

- **LayoutConfig**：`width_px`、`padding`、字体路径与字号、颜色、`line_spacing` 等。
- **RenderSection**（内部）：标题区、结论区、行动项区的矩形与高度 —— 由渲染器根据 `MeetingReport` + `LayoutConfig` 计算，不必持久化。

### 2.3 LLM 输出约定

要求模型输出 **JSON**，字段与 `MeetingReport` 对齐；解析失败时采用**降级策略**（例如将全文作为一条 summary，或重试一次，或在图上标注「自动提取受限」）。

## 三、关键流程（Step by Step）

1. **启动**：解析命令行参数（输入路径、输出 PNG 路径、可选配置）。
2. **读取**：以二进制读入文件 → 解码为 UTF-8（失败则尝试常见编码或明确报错）。
3. **预处理（可选）**：截断过长文本（防止超出模型上下文）、压缩连续空行。
4. **信息抽取**
   - 若启用 LLM：构造 prompt（角色：会议纪要分析；输出：严格 JSON；语言与原文一致或指定为中文）。
   - 调用 API → 解析 JSON → 校验必填语义（至少 `title` 存在，且 `summary` 与 `action_items` 不宜同时为空）。
   - 失败：降级为规则方案，或占位展示并标明「自动提取失败」。
5. **布局规划**：根据行数估算画布高度（或固定高度 + 文末「详见原文」提示）；划分：标题栏 → 关键结论 → 行动项（可选用图标或 emoji 增强可读性，需注意字体支持）。
6. **渲染**：使用 Pillow 新建 RGB 图像 → 分块绘制文字（中文需加载支持中文的字体文件，避免默认字体缺字）→ 按显示宽度自动换行。
7. **输出**：保存为 PNG，设置合适退出码。

## 四、功能模块与可选优化对应关系

| 你的想法 | 对应模块 |
|----------|----------|
| 输入处理（文本/Markdown） | 输入层 + 可选 Markdown 预处理 |
| 信息抽取（规则 / LLM） | 信息抽取层 |
| 报告生成（模板、视觉布局） | 渲染层 + `LayoutConfig` / 多模板 |
| 输出 PNG | 输出层 |
| 多模板、图标/emoji | 多套布局配置；emoji 以 Unicode 绘制（字体需覆盖） |

## 五、落地前可明确的参数

- **是否必须离线**：决定是否强依赖云端 LLM，或需要本地模型/纯规则降级。
- **目标画布宽度**：例如 1080px 竖版，便于统一字号与换行规则。
- **字体文件路径**：生产环境需 bundled 或配置项，保证中英文与 emoji（若需要）显示正常。
