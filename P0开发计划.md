# 智能安全邮箱 — P0 开发计划

**依据**：[P0需求范围.md](./P0需求范围.md)、[P0技术方案.md](./P0技术方案.md)。

**约定**：下文默认 `BASE=http://127.0.0.1:8080`，API 前缀 `/api/v1`；数据库为单文件 SQLite；测试均可在本机终端复现（`curl` + 可选 `sqlite3`）。实现语言不限，模块名与 [P0技术方案.md](./P0技术方案.md) §2 对齐。

---

## 步骤总览

| 步骤 | 里程碑 |
|------|--------|
| 1 | `server` 可启动，库表就绪，健康检查可访问 |
| 2 | `POST /register` 可用，用户落库 |
| 3 | `POST /login` 签发 token，鉴权中间件生效 |
| 4 | `POST /mails` 发信落库 |
| 5 | `GET /mails`、`GET /mails/{id}` 收信 |
| 6 | `client`：`register` / `login`（写 token） |
| 7 | `client`：`send` / `inbox` / `read` |
| 8 | 重启持久化 + 全链路验收 |

## 步骤 1：`server` 骨架 + 数据库迁移 + 健康检查

### 实现内容

- 解析监听地址、SQLite 文件路径（如环境变量 `ADDR`、`DB_PATH`）。
- 启动时打开数据库并执行 **迁移**：创建 `users`、`mails` 表（字段对齐 [P0技术方案.md](./P0技术方案.md) §3.1–3.2；`Attachment` 表 P0 可不建）。
- 提供 **GET** `/health` 或 `/api/v1/health`（无需鉴权），返回固定 JSON，如 `{"ok":true}`。
- 进程启动失败（端口占用、DB 不可写）时退出非 0。

### 需要的代码模块

- `main`、`config`
- `store`：连接、迁移函数
- `transport/http`：注册健康路由、JSON 响应

### 测试方法（可复现）

```bash
# 1）启动（示例；以你实际二进制/参数为准）
export DB_PATH=/tmp/mail-p0-step1.db
rm -f "$DB_PATH"
./server   # 或 go run ./cmd/server

# 2）健康检查
curl -sS "$BASE/health"
# 期望：HTTP 200，体为 JSON 且含 ok 为 true（或与约定一致）

# 3）表存在（可选）
sqlite3 "$DB_PATH" ".schema users"
sqlite3 "$DB_PATH" ".schema mails"
# 期望：能看到 users、mails 的 CREATE TABLE
```

## 步骤 2：注册 `POST /api/v1/register`

### 实现内容

- 请求体：`{"username","password"}`；密码 **bcrypt**（或方案已定算法）哈希后写入 `users`；用户名唯一（建议归一化大小写规则并落库）。
- 响应：201 + `{"user_id":...}`；重复用户名 **409**；统一错误体格式（见技术方案 §4）。

### 需要的代码模块

- `store`：`CreateUser` / 按用户名查询
- `service`：注册校验、哈希
- `handler`：`Register`
- `transport/http`：路由 `POST /api/v1/register`

### 测试方法（可复现）

```bash
curl -sS -o /dev/stderr -w "%{http_code}\n" -X POST "$BASE/api/v1/register" \
  -H "Content-Type: application/json" \
  -d '{"username":"alice","password":"secret1"}'
# 期望：201，响应含 user_id

curl -sS -w "\n%{http_code}\n" -X POST "$BASE/api/v1/register" \
  -H "Content-Type: application/json" \
  -d '{"username":"alice","password":"any"}'
# 期望：409

sqlite3 "$DB_PATH" "SELECT id, username FROM users WHERE username='alice';"
# 期望：一行数据，password 列若为明文则不合格（应只有哈希）
```

## 步骤 3：登录 `POST /api/v1/login` + Bearer 鉴权

### 实现内容

- `POST /api/v1/login`：校验用户名密码，成功返回 `token` + `expires_at`（JWT 或 opaque，与技术方案一致）。
- 增加中间件：除 `register`、`login`、`health` 外，`Authorization: Bearer <token>` 无效则 **401**。
- 可先挂一个 **探测接口**（如 `GET /api/v1/me` 返回当前 `user_id`），用于验证中间件；若直接进入步骤 4，也可在发信接口上首次要求鉴权。

### 需要的代码模块

- `service`：登录、`VerifyPassword`
- `session`：签发/校验 token
- `transport/http`：中间件、`POST /api/v1/login`、（可选）`GET /api/v1/me`

### 测试方法（可复现）

```bash
# 先确保步骤 2 已注册 alice / secret1

TOKEN=$(curl -sS -X POST "$BASE/api/v1/login" -H "Content-Type: application/json" \
  -d '{"username":"alice","password":"secret1"}' | jq -r .token)
test -n "$TOKEN" && echo "token ok"

curl -sS -o /dev/stderr -w "%{http_code}\n" "$BASE/api/v1/me" -H "Authorization: Bearer $TOKEN"
# 期望：200（若实现了 /me）

curl -sS -w "%{http_code}\n" "$BASE/api/v1/me" -H "Authorization: Bearer wrong"
# 期望：401
```

## 步骤 4：发信 `POST /api/v1/mails`

### 实现内容

- 需鉴权；请求体：`to_username`、`subject`、`body`。
- 将 `to_username` 解析为 `to_user_id`，不存在则 **404**；插入 `mails`（`from_user_id` 为当前用户）。
- 响应：201 + `{"mail_id":...}`。

### 需要的代码模块

- `store`：`InsertMail`、按用户名查用户
- `service`：`SendMail`（权限：必须登录；收件人存在性）
- `handler`、`transport/http`：`POST /api/v1/mails`

### 测试方法（可复现）

```bash
# 注册 bob；alice 登录拿到 TOKEN（同步骤 3）
curl -sS -X POST "$BASE/api/v1/register" -H "Content-Type: application/json" \
  -d '{"username":"bob","password":"secret2"}'

TOKEN=$(curl -sS -X POST "$BASE/api/v1/login" -H "Content-Type: application/json" \
  -d '{"username":"alice","password":"secret1"}' | jq -r .token)

curl -sS -w "\n%{http_code}\n" -X POST "$BASE/api/v1/mails" \
  -H "Authorization: Bearer $TOKEN" -H "Content-Type: application/json" \
  -d '{"to_username":"bob","subject":"hi","body":"hello bob"}'
# 期望：201，含 mail_id

curl -sS -w "%{http_code}\n" -X POST "$BASE/api/v1/mails" \
  -H "Authorization: Bearer $TOKEN" -H "Content-Type: application/json" \
  -d '{"to_username":"nobody_x","subject":"x","body":"y"}'
# 期望：404

sqlite3 "$DB_PATH" "SELECT id, from_user_id, to_user_id, subject FROM mails;"
# 期望：一行，收件人为 bob 对应 user_id
```

## 步骤 5：收件箱 `GET /api/v1/mails` + 读信 `GET /api/v1/mails/{id}`

### 实现内容

- `GET /api/v1/mails?limit=&offset=`：当前用户为 **收件人** 的列表；项含 `id`、`from_username`、`subject`、`created_at`（**不含** `body`）。
- `GET /api/v1/mails/{id}`：若当前用户是该邮件收件人，返回含 `body` 的详情；否则 **403**；不存在 **404**。

### 需要的代码模块

- `store`：按 `to_user_id` 列表、按 `id` 取邮件并 join 发件人用户名
- `service`：列表分页、详情鉴权
- `handler`、`transport/http`：两条 GET 路由

### 测试方法（可复现）

```bash
# 延续步骤 4：alice 发给 bob 后
TOKEN_BOB=$(curl -sS -X POST "$BASE/api/v1/login" -H "Content-Type: application/json" \
  -d '{"username":"bob","password":"secret2"}' | jq -r .token)

curl -sS "$BASE/api/v1/mails" -H "Authorization: Bearer $TOKEN_BOB"
# 期望：items 中能看到来自 alice 的一封，含 id

MID=$(curl -sS "$BASE/api/v1/mails" -H "Authorization: Bearer $TOKEN_BOB" | jq -r '.items[0].id')
curl -sS "$BASE/api/v1/mails/$MID" -H "Authorization: Bearer $TOKEN_BOB"
# 期望：body 为 hello bob

curl -sS -w "%{http_code}\n" "$BASE/api/v1/mails/$MID" -H "Authorization: Bearer $TOKEN_ALICE"
# 期望：403（alice 非该封收件人时；TOKEN_ALICE 为 alice 的 token）
```

## 步骤 6：CLI — `register` 与 `login`

### 实现内容

- 子命令：`client register --username ... --password ...`、`client login ...`（或等价交互）。
- `config`：`SERVER_URL` 默认 `http://127.0.0.1:8080`；`login` 成功后将 token 写入约定路径（如 `~/.mail-cli/token` 或 `$TOKEN_FILE`），权限 **0600**。
- `api` 模块封装步骤 2、3 的 HTTP 调用。

### 需要的代码模块

- `cmd` / `main`、`config`、`api`、`ui`（打印成功/失败）

### 测试方法（可复现）

```bash
export SERVER_URL=$BASE
rm -f ~/.mail-cli/token

./client register --username carol --password p3
# 期望：退出码 0，服务端库中有 carol

./client login --username carol --password p3
# 期望：退出码 0，token 文件存在且非空

./client login --username carol --password wrong
# 期望：非 0 退出
```

## 步骤 7：CLI — `send`、`inbox`、`read`

### 实现内容

- `send`：读 token，调用 `POST /api/v1/mails`。
- `inbox`：调用 `GET /api/v1/mails`，表格或列表打印。
- `read <id>`：调用 `GET /api/v1/mails/{id}`，打印正文。

### 需要的代码模块

- `cmd`、`api`、`ui`

### 测试方法（可复现）

```bash
# 终端 1：server 已启动
# 用户 dave / eve 已 register；dave login；eve login

./client login --username dave --password pd
./client send --to eve --subject s1 --body "line1"
./client login --username eve --password pe
./client inbox
./client read 1
# 期望：eve 能看到来自 dave 的邮件且正文一致（read 的 id 以 inbox 输出为准）
```

## 步骤 8：持久化验证 + P0 闭环验收

### 实现内容

- 无新增业务功能；整理 **启动说明**（环境变量、默认端口、DB 路径）。
- （可选）仓库内增加 `scripts/smoke.sh`：顺序执行注册双用户、登录、发信、收信、`read`，失败即非 0 退出——与 [P0需求范围.md](./P0需求范围.md) §4 对齐。

### 需要的代码模块

- 文档或 `scripts/`；代码无强制变更。

### 测试方法（可复现）

```bash
# 1）写入数据后重启 server
pkill -f './server'   # 或你的进程名
./server &
sleep 1

# 2）用原用户名登录，inbox/read 仍一致
./client login --username eve --password pe
./client inbox
# 期望：历史邮件仍在

# 3）可选：跑 smoke 脚本
./scripts/smoke.sh
# 期望：退出码 0
```
